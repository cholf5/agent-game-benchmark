# Slime Mini Game 引擎无关 PRD（含 AI Agent 测试 Rubric）

## 1. 文档目的
本 PRD 用于定义一道引擎无关的小游戏实现题，评估 AI Agent 在不同游戏引擎下的实现能力、工程能力与交付质量。

候选实现引擎包括但不限于 Unity、Unreal、Godot、MonoGame。

## 2. 题目目标
给定一组游戏资源（图像、音频、字体等）与本 PRD，要求 AI Agent 完成一个可运行的 2D 小游戏，核心玩法如下：
- 玩家控制史莱姆移动。
- 场景内有一只会移动的蝙蝠。
- 史莱姆碰到蝙蝠会得分并成长。
- 史莱姆碰墙或撞到自身身体判定失败（Game Over）。
- 支持标题界面、暂停、重试、退出与基础音量设置。

## 3. 范围定义

### 3.1 必须实现（In Scope）
- 标题界面（开始、设置入口）
- 游戏主循环（开始 -> 进行中 -> 暂停/恢复 -> 失败 -> 重试/退出）
- 史莱姆移动与身体增长
- 蝙蝠移动、边界反弹、被吃后重生
- 得分系统
- 失败判定（碰墙 + 撞身体）
- 基础 UI：分数、暂停面板、失败面板、设置面板
- 音频：背景音乐、UI 音效、收集音效、反弹音效

### 3.2 不要求实现（Out of Scope）
- 存档系统
- 关卡编辑器
- 网络联机
- 多语言本地化框架
- 复杂特效管线或后处理高级调参

## 4. 资源包与路径说明

### 4.1 标准资源目录
本题标准资源目录为：`docs/assets/`

已包含原始内容管线文件（含 XML），具体如下：
- `docs/assets/images/atlas.png`
- `docs/assets/images/atlas-definition.xml`
- `docs/assets/images/tilemap-definition.xml`
- `docs/assets/images/background-pattern.png`
- `docs/assets/images/logo.png`
- `docs/assets/audio/theme.ogg`
- `docs/assets/audio/ui.wav`
- `docs/assets/audio/collect.wav`
- `docs/assets/audio/bounce.wav`
- `docs/assets/effects/grayscaleEffect.fx`
- `docs/assets/fonts/04B_30.ttf`
- `docs/assets/fonts/04b_30.fnt`
- `docs/assets/fonts/04B_30.spritefont`
- `docs/assets/fonts/04B_30_5x.spritefont`

### 4.2 图片资源用途
- `images/atlas.png`：角色与 UI 的核心图集。
- `images/atlas-definition.xml`：图集切片与动画定义（权威来源）。
- `images/tilemap-definition.xml`：地牢房间 tilemap 布局定义（权威来源）。
- `images/background-pattern.png`：标题背景平铺滚动纹理。
- `images/logo.png`：可用于标题页 Logo（可选）。

### 4.3 动画播放规范（基于 XML）
- 必须从 `images/atlas-definition.xml` 读取动画定义并还原到目标引擎动画系统。
- 至少应实现并正确播放：
  - `slime-animation`（史莱姆）
  - `bat-animation`（蝙蝠）
  - 按钮状态动画（若 XML 中定义了按钮 idle/hover/pressed 序列）
- 动画要求：
  - 循环动画需循环播放（角色待机/移动类）。
  - 一次性动画（若存在）播放结束后回到默认状态。
  - 不允许把动画“烘焙成单帧静态图”替代实现。

### 4.4 Tilemap 使用规范（基于 XML）
- 必须以 `images/tilemap-definition.xml` 作为房间布局来源。
- 要求可复现“房间边界可碰撞、内部可移动”的空间语义。
- tile 尺寸、缩放、碰撞边界可做引擎适配，但行为结果必须一致：
  - 史莱姆头部越过房间边界判定失败。
  - 蝙蝠撞边界发生反弹。

### 4.5 音频资源用途
- `audio/theme.ogg`：背景音乐（循环）。
- `audio/ui.wav`：UI 操作反馈（按钮点击、确认等）。
- `audio/collect.wav`：史莱姆吃到蝙蝠。
- `audio/bounce.wav`：蝙蝠碰墙反弹。

### 4.6 字体资源用途
- `fonts/04B_30.ttf`：主字体源文件（推荐优先使用）。
- `fonts/04b_30.fnt` 与 `.spritefont`：原始项目字体描述，可作为字号、风格和布局参考。

### 4.7 资源导入与替代规则
- 首选使用 `docs/assets` 原始资源完成实现。
- 允许因引擎导入限制做“格式层”转换（例如切图、音频导入参数调整），但需满足：
  - 视觉与行为等价。
  - README 记录转换步骤。
- 若某资源确实无法直接使用，可使用等价占位资源，但必须在 README 明确列出差异与原因。

## 5. 核心玩法规格

### 5.1 玩家控制
- 输入语义：上、下、左、右、确认、暂停。
- 推荐默认映射：
  - 键盘：方向键 / WASD，Enter 确认，Esc 暂停
  - 手柄：D-Pad 或左摇杆方向，A 确认，Start 暂停
- 方向切换应具备离散动作感，避免同帧多次冲突输入导致状态抖动。

### 5.2 游戏对象与行为
- 史莱姆：
  - 由“头部 + 身体段”构成。
  - 吃到蝙蝠后身体段增长。
  - 与自身身体发生碰撞时立即失败。
- 蝙蝠：
  - 在房间内移动。
  - 碰到边界发生反弹并触发反弹音效。
  - 被史莱姆吃到后，重新生成到与史莱姆保持安全距离的位置，并重置/随机速度。

### 5.3 碰撞与边界
- 游戏房间存在明确边界。
- 史莱姆头部超出边界即失败。
- 史莱姆头部与蝙蝠碰撞：得分 + 蝙蝠重生 + 史莱姆成长。
- 蝙蝠超出边界应被修正并反弹，不允许持续越界。

### 5.4 分数规则
- 每次吃到蝙蝠固定 +100 分。
- 分数实时显示在 HUD。
- 重试开局时分数归零。

### 5.5 状态机
必须具备以下状态与合法迁移：
- `Title` -> `Playing`
- `Playing` <-> `Paused`
- `Playing` -> `GameOver`
- `GameOver` -> `Playing`（重试）
- `GameOver` -> `Title`（退出）

非法状态跳转（例如 Paused 直接进入 Title 且无用户触发）应避免。

## 6. UI/UX 规格

### 6.1 标题界面
- 显示游戏标题。
- 提供 `Start` 与 `Options`。
- `Start` 进入游戏。
- `Options` 进入设置界面。

### 6.2 设置界面
- 至少包含：
  - 音乐音量滑条（0~1）
  - 音效音量滑条（0~1）
  - 返回按钮
- 滑条调整应立即生效或在确认操作后生效，但行为需在文档中说明。

### 6.3 游戏内界面
- 常驻显示分数文本。
- 暂停时显示暂停面板，支持继续/退出。
- 失败时显示失败面板，支持重试/退出。

### 6.4 交互反馈
- 按钮点击有视觉状态变化（至少 Hover/Pressed 二者之一）。
- 关键 UI 操作播放统一 UI 音效。

## 7. 视觉与音频规格

### 7.1 视觉
- 使用给定资源或等价占位资源。
- 角色动画使用引擎原生 2D 动画机制。
- 可选实现“暂停/失败灰度过渡”效果；若实现应有平滑衰减。

### 7.2 音频
- 背景音乐循环播放。
- 音效类别至少区分：UI、收集、反弹。
- 受设置面板音量参数控制。

## 8. 技术约束（引擎无关）
- 必须使用目标引擎原生 UI 系统，不得以自绘文本+手写命中框替代完整 UI 交互。
- 必须使用目标引擎原生动画系统，不得完全依赖逐帧手写渲染逻辑模拟 UI/角色动画。
- 必须使用目标引擎原生碰撞/物理能力或其标准碰撞组件，不得把所有碰撞都退化为纯手写数学且绕开引擎能力。
- 代码需模块化，至少拆分：输入、玩法逻辑、UI 控制、音频控制。

## 9. 交付物要求
- 可运行工程目录（含依赖声明与版本说明）。
- `README`（启动步骤、输入说明、已知问题、配置说明）。
- 关键模块说明文档（可并入 README）：
  - 状态机实现方式
  - 碰撞方案
  - 资源导入/绑定方案
- 可选：短视频或 GIF 展示完整流程。

## 10. 验收标准（功能）
验收时至少验证以下场景：
- 从标题界面进入游戏成功。
- 史莱姆可移动并稳定响应输入。
- 吃到蝙蝠时分数 +100 且史莱姆增长。
- 撞墙触发失败，失败面板出现。
- 暂停与恢复状态正确切换。
- 重试后分数清零、对象重置。
- 设置面板可调节音乐与音效音量并生效。

## 11. 非功能要求
- 桌面环境普通硬件下稳定运行，无明显卡顿。
- 无阻塞主循环的高频日志或低效轮询。
- 关键代码具备可读命名与基本注释。
- 出错场景（资源缺失、配置不完整）有清晰提示，不得静默失败。

## 12. 统一评分 Rubric（跨引擎）
总分 100，建议分配如下：

### 12.1 玩法正确性（40 分）
- 核心循环完整（10）
- 碰撞与失败判定正确（10）
- 分数与成长机制正确（10）
- UI 流程与交互一致（10）

### 12.2 工程质量（30 分）
- 模块划分清晰（10）
- 代码可维护性（命名、结构、复用）（10）
- 资源与配置组织合理（10）

### 12.3 文档与可交接性（20 分）
- README 完整可复现（10）
- 技术说明清晰（状态机/碰撞/资源绑定）（10）

### 12.4 性能与稳定性（10 分）
- 无明显性能问题（5）
- 边界场景稳定（重复暂停/重试、多次碰撞）（5）

## 13. 扣分项（建议）
- 使用非原生 UI/动画/碰撞方案替代核心要求：每项 -5 到 -15。
- 状态机混乱导致流程错误：-10。
- 无法按 README 启动：-20（可直接判不通过）。
- 关键功能缺失（暂停、失败、重试任一缺失）：每项 -10。

## 14. 测试执行建议（面向出题人）
- 同一套 PRD 与资源包发给多个 Agent。
- 指定统一时间盒（例如 90~180 分钟）。
- 要求提交：
  - 工程文件
  - README
  - 变更说明
- 使用 Rubric 同步打分，并记录“首次可运行时间”“修复轮次”等过程指标。

## 15. 引擎附录模板（可选）
出题时可附上“引擎适配约束页”（一页纸）：
- 指定引擎版本范围
- 指定允许/禁止插件范围
- 指定打包目标平台
- 指定输入系统选择（新旧输入系统等）

该附录不改变核心玩法要求，仅用于保障各引擎评测公平性与可执行性。
